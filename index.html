Source — A developer pushes code or opens a pull request on GitHub. A configured webhook fires an HTTP POST to Jenkins, kicking off the pipeline automatically. Build & Test (Jenkins) — Jenkins orchestrates three sequential stages: Checkout (clones the repo), Build (compiles, lints, installs dependencies), and Test (unit and integration tests). If any stage fails, the pipeline short-circuits and sends a notification — shown by the red dashed failure path looping back. Package (Docker) — On a clean test pass, Jenkins runs docker build using your Dockerfile to produce an image, then pushes it to a registry (Docker Hub, ECR, or a private registry). The image is tagged with the commit SHA or build number for traceability. Deploy (EC2) — Jenkins SSHs into the target EC2 instance (or uses SSM/Ansible), pulls the new image from the registry, and runs it via docker run or docker-compose up. The app is now live. A few things worth knowing about this pattern: Jenkinsfile lives in the repo itself (pipeline as code), so the pipeline definition is version-controlled alongside the application. IAM roles on EC2 should grant the instance permission to pull from ECR — avoid baking credentials into the instance or the image.
